# RailsHx ActionCable Guide

RailsHx ActionCable support is a typed Haxe authoring layer for normal Rails
channels and subscriptions. The compiler emits `ActionCable::Channel::Base`
subclasses and JavaScript client code calls the standard ActionCable
`consumer.subscriptions.create(...)` API.

## Server Channels

Annotate a Haxe class with `@:railsChannel` and extend
`rails.action_cable.Channel<TParams, TPayload>`:

```haxe
import rails.ActionCable;
import rails.action_cable.Channel;
import rails.action_cable.Stream;
import rails.action_cable.SubscriptionParam;

typedef TodoSubscriptionParams = {
	var listId:String;
}

typedef TodoBroadcast = {
	var title:String;
	var completed:Bool;
}

class TodoCable {
	public static inline function listId():SubscriptionParam<String> {
		return SubscriptionParam.named("listId");
	}

	public static inline function listStream(listId:String):Stream<TodoBroadcast> {
		return Stream.named("todos:" + listId);
	}
}

@:railsChannel
class TodosChannel extends Channel<TodoSubscriptionParams, TodoBroadcast> {
	public function subscribed():Void {
		var listId = param(TodoCable.listId());
		streamFrom(TodoCable.listStream(listId));
	}

	public function unsubscribed():Void {
		stopAllStreams();
	}

	public static function announce(listId:String, title:String):Void {
		ActionCable.broadcast(TodoCable.listStream(listId), {
			title: title,
			completed: false
		});
	}
}
```

Generated Ruby stays Rails-native:

```ruby
class TodosChannel < ActionCable::Channel::Base
  def subscribed
    list_id = params["list_id"]
    stream_from "todos:#{list_id}"
  end

  def unsubscribed
    stop_all_streams
  end

  def self.announce(list_id, title)
    ActionCable.server.broadcast "todos:#{list_id}", { title: title, completed: false }
  end
end
```

The exact temporary variable names may differ, but the important contract is the
same: Rails owns channel runtime behavior.

## Type Safety

`SubscriptionParam<T>` keeps channel params typed at the Haxe boundary. A
`SubscriptionParam<String>` cannot be assigned to an `Int` without a compile
error. Plain strings do not satisfy `param(...)`; define shared keys with
`SubscriptionParam.named(...)`.

`Stream<TPayload>` carries the broadcast payload shape. If
`TodoCable.listStream(...)` returns `Stream<TodoBroadcast>`, then
`ActionCable.broadcast(...)` must receive a `TodoBroadcast` payload. Missing
fields such as `completed` fail during Haxe compilation. Plain strings do not
satisfy `streamFrom(...)` or `ActionCable.broadcast(...)`; use
`Stream.named(...)` at a typed boundary.

`@:railsChannel` also requires an instance `subscribed()` method so an empty
channel does not silently compile as a broken Rails channel.

## Client Subscriptions

Haxe-authored JavaScript can subscribe through a typed facade while keeping the
Rails-generated consumer module:

```haxe
import rails.action_cable.Consumer;

var consumer = Consumer.create();
var subscription = Consumer.subscribe(consumer, "Channels::TodosChannel", {listId: "open"}, {
	connected: function():Void {},
	disconnected: function():Void {},
	received: function(payload:TodoBroadcast):Void {
		trace(payload.title);
	}
});
subscription.perform(TodoCable.pingAction(), {title: "client ping"});
```

`Consumer.create()` returns a typed `Consumer` handle and lowers to
`ActionCable.createConsumer()`. `Consumer.subscribe(...)` requires that typed
handle, so raw objects such as `{}` do not accidentally satisfy the ActionCable
client boundary. The subscription call still lowers to
`consumer.subscriptions.create(...)`. Rails apps can keep the standard
`app/javascript/channels/consumer.js` generated by Rails and pass that typed
consumer into compiled Haxe client code.

Client-side channel actions use `Action<TData>` tokens:

```haxe
import rails.action_cable.Action;

typedef TodoPingPayload = {
	var title:String;
}

class TodoCable {
	public static inline function pingAction():Action<TodoPingPayload> {
		return Action.named("ping");
	}
}
```

`subscription.perform(TodoCable.pingAction(), {title: "client ping"})` lowers to
Rails' normal `subscription.perform("ping", { ... })`, but raw action strings do
not satisfy the typed default and missing payload fields fail during Haxe
compilation. Use `subscription.performUnchecked("legacy_action", rawPayload)`
only for reviewed interop with Rails channels that do not have a typed action
contract yet.

## Tests

Use the static ActionCable smoke for compiler and generated-output coverage:

```bash
npm run test:action-cable
```

The smoke compiles the server channel, syntax-checks generated Ruby, compiles
the Haxe JS client, and verifies negative Haxe typing cases for invalid payloads,
wrong param assignment, raw consumer objects, raw perform action strings, invalid
perform payloads, and missing `subscribed()`.

When Rails gems are available, the same smoke materializes a generated Rails app
and runs `ActionCable::Channel::TestCase` against the generated channel. The
runtime pass asserts subscription confirmation, typed stream subscription,
`perform :ping` transmissions, and broadcast payloads. Local
`npm run test:action-cable` skips this runtime pass if the generated app bundle
is unavailable; `npm run test:rails-runtime` includes
`REQUIRE_RAILS=1 npm run test:action-cable` and makes missing runtime
dependencies fail closed.
