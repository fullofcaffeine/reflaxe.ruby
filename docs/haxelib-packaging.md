# Haxelib Packaging

`reflaxe.ruby` should behave like a normal compiler created by
`haxelib run reflaxe new` unless a Ruby/Rails release asset requires a narrow
extension.

## Source Layout

The source checkout follows Reflaxe's generated project model:

- `src/**` contains compiler code, macros, and Ruby AST/lowering support.
- `std/**` contains additive Ruby/Rails Haxe APIs such as `ruby.*`,
  `rails.*`, and `devisehx.*`.
- `std/ruby/_std/**/*.hx` contains Ruby-owned replacements for upstream Haxe
  stdlib modules.

`haxelib.json` is the source of truth for that layout:

```json
"reflaxe": {
  "name": "Ruby",
  "abbv": "rb",
  "stdPaths": [
    "std",
    "std/ruby/_std"
  ]
}
```

Repo-local Ruby target builds use `haxe_libraries/reflaxe.ruby.hxml` for
`src/`, `std/`, `std/ruby/_std/`, vendored Reflaxe, and compiler init macros.
This mirrors the source-mode setup used by the sibling Reflaxe compilers:
source checkouts see `_std`, while published haxelib packages receive
generated `.cross.hx` files from Reflaxe build.

RailsHx browser builds do not use `-lib reflaxe.ruby`. They use
`-lib railshx.client`, whose hxml includes only the browser-safe/shared `std/`
surface and no Ruby compiler macros, vendored Reflaxe classpath, or
`std/ruby/_std` overrides. The shared `std/` surface intentionally keeps typed
Turbo tokens such as `StreamName`, `StreamTarget`, and `TurboStreamAction`
available to both server and browser code; moving those into a client-only
tree would break existing compiler lowerings such as typed ActionView Turbo
streams. Browser-only APIs such as `rails.turbo.Turbo` and
`reflaxe.js.Async` also live in `std/` today, but the library boundary keeps
them out of Ruby target bootstrap behavior.

The canonical browser lane also uses `-lib genes`. Haxe still types a normal
`-js` build, while Genes registers a custom final emitter that produces split
ES modules for Rails importmap/Propshaft. See
[RailsHx Client JavaScript And Genes](railshx-client-javascript.md) for why the
stock Haxe JavaScript emitter is not the generated-app default and how the
packaged client module graph reaches Rails.

Do not check in generated `src/**/*.cross.hx` mirrors for those `_std` files.

## Package Layout

Release zips delegate the generic package layout to Reflaxe's build runner:

- Reflaxe copies the package `classPath` (`src`) into `_Build/src`.
- Reflaxe merges each `reflaxe.stdPaths` entry into `_Build/src`.
- Reflaxe converts files copied from a path ending in `_std` into packaged
  `.cross.hx` files, for example `std/ruby/_std/Std.hx` becomes
  `src/Std.cross.hx`.
- Reflaxe sanitizes the packaged `haxelib.json` by removing the `reflaxe`
  metadata field.

That means source checkouts and installed haxelib packages intentionally look
different:

```text
source:  std/ruby/_std/haxe/Json.hx
package: src/haxe/Json.cross.hx
```

Ruby target consumers of an installed haxelib package should use only:

```hxml
-lib reflaxe.ruby
```

They should not add `std/` or `std/ruby/_std/` manually.

The current release workflow distributes this Haxelib-compatible ZIP only as
a GitHub Release asset; it does not publish to the Haxelib registry. Consumers
download the versioned ZIP and its SHA-256 identity sidecar, verify the bytes,
tag, and source SHA as documented in
[Reproducible Release Artifacts](release-artifacts.md), and then install that
local ZIP. A future registry lane must publish the exact already-verified bytes
rather than rebuild a second package.

RailsHx browser/client consumers should use the app-local `railshx.client`
hxml generated by the `hxruby` gem:

```hxml
-lib railshx.client
```

Do not point browser builds at the flattened haxelib package `src/` directory:
that directory contains Ruby-target `.cross.hx` std overrides by design.

## Ruby Wrapper

The standard Reflaxe references in `../haxe.compilerdev.reference` mostly run
`haxelib run reflaxe build` directly. Ruby keeps a thin wrapper at
`scripts/release/build-haxelib-package.js` because the haxelib release also
ships assets that generic Reflaxe does not know about:

- `runtime/hxruby/**` Ruby runtime helpers copied or required by generated Ruby.
- `lib/**` and `hxruby.gemspec` for the Ruby gem bridge.
- `vendor/reflaxe/**` so installed `reflaxe.ruby` can resolve the patched
  Reflaxe framework used by this compiler.
- `vendor/genes/**` for RailsHx client builds generated by the `hxruby` gem.
- `docs/**`, `examples/**`, `README.md`, `CHANGELOG.md`, and `LICENSE`.

The wrapper must not reimplement Reflaxe's std/classpath flattening. Its job is
only:

1. Stage the source files Reflaxe needs.
2. Run `haxe -cp vendor/reflaxe --run Run build _Build --deleteOldFolder`.
3. Copy Ruby-specific release extras into `_Build`.
4. Zip `_Build`.

`npm run test:haxelib-package` enforces this boundary by checking for packaged
`src/**/*.cross.hx` files, rejecting top-level `std/**` and
`haxe_libraries/**` leakage, verifying that packaged `haxelib.json` was
sanitized, and compiling installed Ruby-target consumers through
`-lib reflaxe.ruby`. Browser/client compilation is covered by
`npm run test:gem-package` and generated-app smokes through `railshx.client`,
because the haxelib package's flattened `.cross.hx` sources are intentionally
Ruby-target only.

## Vendored Reflaxe

Ruby currently follows the `haxe.rust` precedent and vendors Reflaxe, including
`vendor/reflaxe/Run.hx`, so release packaging is deterministic and does not
depend on a developer's global `haxelib dev reflaxe` state. This is especially
important because the local environment can resolve `haxelib run reflaxe` to a
different sibling checkout.

The vendor currently includes the lazy function-field metadata fix proposed in
[Reflaxe PR #52](https://github.com/SomeRanDev/reflaxe/pull/52), patch commit
`024937acffd242f129265d969a840d3779f02bcd`. Haxe can represent a method loaded
late through `Context.getType` as `TLazy(TFun(...))`; the patch forces only the
lazy wrapper before Reflaxe extracts function metadata and constructs overload
cache keys. It deliberately preserves typedef and abstract identity. Detailed
provenance, the unsafe alternatives it avoids, and the removal rule live in
`vendor/reflaxe/PATCHES.md`.

`npm run test:reflaxe-lazy-function-field` executes that late-loaded type path
through a real Reflaxe `filterTypes` callback. The release contract also
requires the patched `ClassFieldHelper.hx` and `PATCHES.md` to be present in the
haxelib archive. Once an official pinned Reflaxe release contains PR #52, keep
the regression, update the baseline, and remove the redundant vendor delta.

This vendoring is pragmatic, not a permanent design goal. A future cleanup may
remove the vendored Reflaxe copy and depend on a normal released `reflaxe`
haxelib once these conditions are true:

- Ruby no longer needs local patches to Reflaxe framework sources.
- CI and release jobs can resolve the exact required Reflaxe version
  deterministically.
- `haxelib run reflaxe build` works without sibling-checkout or global-dev
  ambiguity.
- Installed `reflaxe.ruby` consumers still compile without requiring extra
  manual classpaths.

If that migration happens, keep the package shape and tests unchanged. Only the
source of the Reflaxe build runner should change.
