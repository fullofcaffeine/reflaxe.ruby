# RailsHx Generated Artifact Ownership

RailsHx writes into Rails applications, so it must distinguish generated files
from app-owned Rails files. The default is fail-closed: if RailsHx cannot prove a
file is RailsHx-owned, it refuses to overwrite it.

## Ownership Signals

RailsHx recognizes generated ownership through either signal:

- `.railshx/manifest.json` contains an entry for the output path.
- The existing file starts with `# Generated by RailsHx.`

Generator-owned Haxe files are usually tracked by manifest rather than a header,
because comments at the top of every generated `.hx` source can get noisy.
Rails-facing generated artifacts such as migrations, generated tests, and
compiler-emitted Rails templates should keep explicit headers where practical
because Rails developers inspect those files directly.

## Manifest Shape

The manifest is app-relative and committed when generated artifacts are
committed:

```json
{
  "version": 1,
  "outputs": [
    {
      "output": "src_haxe/models/Todo.hx",
      "kind": "haxe_source",
      "source": "hxruby:scaffold",
      "sha256": "..."
    }
  ]
}
```

The `output` is always app-relative. The `kind` describes the artifact class
(`haxe_source`, `haxe_migration_source`, `route_extern`, `client_js`,
`rails_config`, and so on). The `source` records the generator or compiler lane
that produced the file. The checksum is over the written file content.

## Write Policy

Normal generator writes follow this rule:

1. If the output path is absent, write the file and record it in the manifest.
2. If the output path exists and is manifest-owned, rewrite it and update the
   checksum.
3. If the output path exists and starts with a RailsHx generated header, rewrite
   it and record it.
4. If the output path exists but is not RailsHx-owned, fail with an actionable
   overwrite diagnostic.
5. If `--force` is explicit, overwrite and take RailsHx ownership.

`--force` is intentionally loud. It is for repair or deliberate ownership
transfer, not the normal path.

## Rails-Owned Files

These files remain Rails/app-owned unless a manifest entry or generated header
proves RailsHx owns the exact path:

- `db/migrate/**/*.rb`
- `app/views/**/*.erb`
- `app/models/**/*.rb`
- `app/controllers/**/*.rb`
- `config/routes.rb`
- Rails asset and JavaScript files
- Existing Ruby tests

Generators may create Haxe source that later compiles to these Rails artifacts,
but RailsHx should not overwrite hand-written Rails files as a side effect.

## Compiler Output

The compiler should continue emitting Rails-native generated headers for
migrations, Rails tests, HHX/ERB artifacts, and initializers. Ruby-native Rails
generators use the manifest-backed writer. Compiler-side manifest integration is
being added per artifact family: Haxe-owned `@:railsRoutes` output now records
both `config/routes.rb` and `.railshx/routes.haxe.json` in `.railshx/manifest.json`.
The route-specific manifest keeps the Haxe route declaration tree for future
Rails parity checks; raw `uncheckedRubyRoute(...)` entries are marked opaque and
record only a checksum of the raw line.

Deeper manifest integration still belongs at the Reflaxe output boundary so
every `setExtraFile` artifact can eventually be recorded consistently.

## Tests

Generator smoke tests should cover:

- Manifest creation on first write.
- Safe rewrite of manifest-owned files.
- Refusal to overwrite a hand-written file at the same path.
- Safe relative path enforcement.
- `--force` behavior only where the public generator exposes it.

Compiler surfaces should keep snapshot tests for generated output shape and
negative tests for unsafe paths or duplicate generated files.

For routing, `examples/rails_routes_dsl` is the focused compiler fixture. Its
snapshots cover both generated `config/routes.rb` and the route-specific
`.railshx/routes.haxe.json` manifest, while `examples/todoapp_rails` proves the
same ownership model in a running Rails app.
