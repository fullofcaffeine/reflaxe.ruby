# Debugging Generated Applications

RubyHx and RailsHx keep debugging deliberately close to the target runtime.
They do not add a second server runtime or a Ruby source-map subsystem.

## Source Contract

Use the source location that owns each stage:

- A Haxe compiler or RailsHx macro error points to the Haxe or HHX source that
  must change.
- A Ruby or Rails server backtrace points to generated Ruby or ERB. That exact
  generated file and line is the supported line-level debugging source for
  server failures in 1.0.
- A browser stack trace uses the source map emitted by the Haxe client build to
  correlate generated JavaScript with Haxe source.

Server backtraces are not currently remapped to Haxe or HHX lines, and
`haxe.CallStack` does not promise normalized Ruby frames. Generated files are
intended to stay readable and deterministic so normal Ruby and Rails tools remain
useful. Ruby/Rails debugging knowledge is still part of operating the result.

## Server Workflow

Start with the non-mutating health report:

```bash
bundle exec rake hxruby:doctor
```

It reports the installed `hxruby`, Ruby, Haxe, Node, and loaded Rails versions;
the generated Ruby roots read from the server HXML; and the ownership manifest
version, output count, and generator source labels. It also reports missing
outputs and checksum drift. Manifest `source` labels identify the generator or
ownership path, not a Haxe file-and-line mapping.

Then rebuild and syntax-check the generated server output:

```bash
bundle exec rake hxruby:check
bundle exec rake hxruby:check CLIENT=1 ROUTES=1 ZEITWERK=1
```

When Rails still fails:

1. Read the first relevant application frame in the Ruby/Rails backtrace.
2. Inspect that generated Ruby or ERB file at the reported line.
3. Check its `# Generated by RailsHx.` header and `.railshx/manifest.json` entry
   to distinguish RailsHx-owned output from Rails-owned adoption code.
4. Change the owning Haxe/HHX or Rails-owned source, regenerate, and rerun the
   normal Rails request, job, mailer, test, or production command.

`HxException` preserves the Haxe value while remaining a Ruby `StandardError`.
Ruby raise-site backtraces and native exception causes remain available, so job
retry logs and Rails error reports retain their normal diagnostic shape.

## Views And Assets

RailsHx-owned HHX compiles to ordinary ERB. A render failure therefore reports
the generated template path and line; inspect that ERB, then change the owning
HHX. Existing Rails-owned ERB remains its own source of truth.

For browser or production asset failures, run the client and production gates
separately so the failing owner is clear:

```bash
bundle exec rake hxruby:compile:client
RAILS_ENV=production bundle exec rake hxruby:production
```

Client HXML files generated by RailsHx enable `-D source-map`. The maintained
Playwright and production lanes exercise the same importmap/Propshaft output,
while browser developer tools consume the emitted maps for Haxe correlation.
