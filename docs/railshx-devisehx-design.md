# DeviseHx Design

DeviseHx is the reusable RailsHx companion layer for the Ruby Devise gem. It is
not an authentication runtime. Devise, Warden, Rails routes, Rails controllers,
Rails mailers, migrations, and Bundler remain the runtime source of truth.
DeviseHx adds typed Haxe contracts, deterministic discovery, generated
app-local scope contracts, HHX helpers, and CI validation around normal
Rails/Devise artifacts.

This design folds in the GPT 5.5 Pro review required by
`haxe.ruby-bjv.6.4`.

## Verdict

Use a three-layer model:

- **Reusable package:** `devisehx` / `hxruby-devise` provides typed Haxe
  abstractions for scopes, modules, controller helpers, lifecycle filters,
  mappings, route declarations, parameter sanitizer helpers, mailer/test-helper
  seams, and HHX auth helpers.
- **App-local generated contracts:** each Rails app gets concrete Haxe contracts
  such as `app.auth.UserAuth`, generated from deterministic inventory:
  installed Devise version, scopes, models, routes, schema, initializer, custom
  controllers, and ownership metadata.
- **Rails-owned runtime artifacts:** `Gemfile`, `Gemfile.lock`,
  `config/initializers/devise.rb`, `config/routes.rb` in adoption mode, Devise
  migrations, Ruby models/controllers/mailers/views, and Rails tests stay owned
  by Rails unless explicitly generated under RailsHx manifest ownership.

The invariant is simple: Rails is the runtime oracle; RailsHx is the typed
authoring and validation layer.

## Runtime Ownership

DeviseHx must emit or consume ordinary Rails/Devise constructs:

- `devise :database_authenticatable, ...`
- `devise_for :users`
- `before_action :authenticate_user!`
- `current_user`, `user_signed_in?`, `sign_in`, `sign_out`
- Devise controllers, mailers, route helpers, parameter sanitizer, and test
  helpers

Do not reimplement Warden, Devise mappings, password flows, session storage,
mailers, or route helper naming in Haxe. Rails/Devise owns those semantics.

## Public Haxe API Shape

The central type should be a generated scope token that couples a Rails scope,
route resource, and model type:

```haxe
package app.auth;

import devisehx.Auth;
import devisehx.DeviseScope;
import models.User;

// Generated by DeviseHx from Bundler, Devise mappings, routes, model source,
// and schema. Runtime owner: Devise/Rails. Haxe owner: typed contract only.
final class UserAuth {
	public static final scope:DeviseScope<User> = DeviseScope.of("user", "users", User);
	public static final authenticate = Auth.require(scope);

	public static inline function current(controller:rails.action_controller.Base):Null<User> {
		return Auth.current(controller, scope);
	}

	public static inline function currentRequired(controller:rails.action_controller.Base):User {
		return Auth.currentRequired(controller, scope);
	}

	public static inline function signedIn(controller:rails.action_controller.Base):Bool {
		return Auth.signedIn(controller, scope);
	}

	public static inline function signIn(controller:rails.action_controller.Base, user:User):Void {
		Auth.signIn(controller, scope, user);
	}

	public static inline function signOut(controller:rails.action_controller.Base):Void {
		Auth.signOut(controller, scope);
	}
}
```

Controller usage should integrate with the existing RailsHx lifecycle DSL:

```haxe
@:railsController
class DashboardController extends ApplicationController {
	static final lifecycle = {
		beforeAction(UserAuth.authenticate);
	};

	public function index():Void {
		final user = UserAuth.currentRequired(this);
		render(DashboardView.render({currentUser: user}));
	}
}
```

Generated Ruby should remain familiar:

```ruby
class DashboardController < ApplicationController
  before_action :authenticate_user!

  def index
    user = current_user
    render(...)
  end
end
```

Nullability rule:

- `current(...)` returns `Null<T>`.
- `currentRequired(...)` is allowed when the action is protected by the matching
  auth filter, or when the app deliberately accepts a runtime assertion
  boundary. Strict compile-time flow checks can harden this after the first
  implementation slice.

## Model Modules

Haxe-owned Devise models should use metadata that emits Devise's normal class
macro:

```haxe
import devisehx.model.DeviseModule.*;

@:railsModel("users")
@:devise(UserAuth.scope, [
	databaseAuthenticatable,
	registerable,
	recoverable,
	rememberable,
	validatable
])
class User extends rails.active_record.Base<User>
	implements devisehx.model.DeviseResource<User> {
	@:deviseAuthKey public var email:String;
	public var encryptedPassword:String;
}
```

Generated Ruby:

```ruby
class User < ApplicationRecord
  devise :database_authenticatable,
         :registerable,
         :recoverable,
         :rememberable,
         :validatable
end
```

Rails-owned existing models should use generated extern contracts instead of
pretending Haxe owns the Ruby source:

```haxe
@:native("User")
@:deviseExtern(UserAuth.scope, [
	databaseAuthenticatable,
	registerable,
	recoverable,
	rememberable,
	validatable
])
extern class User extends rails.active_record.Base<User>
	implements devisehx.model.DeviseResource<User> {
	public var email:String;
	public function validPassword(password:String):Bool;
}
```

Known Devise modules should be represented as typed specs:

```haxe
enum DeviseModuleSpec {
	databaseAuthenticatable(?options:DatabaseAuthenticatableOptions);
	registerable(?options:RegisterableOptions);
	recoverable(?options:RecoverableOptions);
	rememberable(?options:RememberableOptions);
	validatable(?options:ValidatableOptions);
	confirmable(?options:ConfirmableOptions);
	lockable(?options:LockableOptions);
	trackable;
	timeoutable(?options:TimeoutableOptions);
	omniauthable(options:OmniauthableOptions);
	unsafeCustom(name:String);
}
```

Schema checks should fail closed for known module requirements:

| Module | Required deterministic checks |
| --- | --- |
| `database_authenticatable` | auth key column, usually `email`; `encrypted_password` |
| `recoverable` | `reset_password_token`, `reset_password_sent_at` |
| `rememberable` | `remember_created_at` |
| `confirmable` | `confirmation_token`, `confirmed_at`, `confirmation_sent_at`; optional `unconfirmed_email` |
| `lockable` | `failed_attempts`, `unlock_token`, `locked_at` depending options |
| `trackable` | sign-in count/time/ip columns |
| `omniauthable` | literal providers and callback routes; provider model fields require explicit review |

## Routes

Devise route declarations may be Rails-owned or Haxe-owned, but route helper
contracts always come from `rails routes`.

Initial DeviseHx should prefer Rails-owned routes:

```ruby
Rails.application.routes.draw do
  devise_for :users
  root "dashboard#index"
end
```

Then:

```bash
bundle exec rake hxruby:routes
```

The route generator has fixture coverage for the normal Devise helper matrix:
sessions, passwords, registrations, confirmations, unlocks, and OmniAuth
authorize/callback helpers. These helpers are generated from Rails route output,
not guessed by DeviseHx.

Haxe-owned Devise route DSL supports a deliberately narrow typed subset:
`DeviseRoutes.deviseFor(UserAuth.scope)` plus `only`/`skip` route-group
selection.

```haxe
import app.auth.UserAuth;
import app.auth.AdminAuth;
import devisehx.routes.DeviseRouteGroup.*;
import devisehx.routes.DeviseRoutes;
import rails.macros.RoutesDsl.*;

@:railsRoutes
class AppRoutes {
	static final routes = {
		DeviseRoutes.deviseFor(UserAuth.scope);
		DeviseRoutes.deviseFor(AdminAuth.scope, { only: [Sessions] });
		DeviseRoutes.deviseFor(AdminAuth.scope, { skip: [Sessions] });
		root(to(DashboardController, index));
	};
}
```

The macro accepts only direct generated scope fields carrying
`@:deviseHxRoute(...)` metadata, then emits ordinary Rails:

```ruby
devise_for :users
devise_for :admins, only: [:sessions]
devise_for :admins, skip: [:sessions]
```

This is intentionally not a string API. Literals, locals, function-produced
scopes, constructed `DeviseScope` values, nested declarations, raw strings,
unknown options, and combined `only` + `skip` fail closed. Duplicate mappings
are allowed only when every duplicate declaration is explicitly split with typed
`DeviseRouteGroup` tokens. That preserves the key ownership boundary: the Haxe
macro declares checked Devise mappings; Devise expands them; Rails names the
helpers; RailsHx consumes the actual `rails routes` output to generate typed
route externs.

Custom controllers, `path`, `path_names`, `class_name`, `singular`, constraints,
and other advanced Devise route options remain Rails-owned until DeviseHx has
typed contracts for them. They should still be consumed through generated route
externs.

## HHX Helpers

Greenfield DeviseHx views should use HHX. Existing Devise ERB views remain
Rails-owned until explicitly adopted.

Reusable helpers should include:

- `AuthLinks.signInLink(scope, ?options)`
- `AuthLinks.signOutButton(scope, ?options)`
- `AuthLinks.signUpLink(scope, ?options)`
- `AuthLinks.editRegistrationLink(scope, ?options)`
- `DeviseFlash.render(...)`
- `DeviseErrors.forResource(resource)`
- typed Devise form helpers for sessions, registrations, passwords,
  confirmations, and unlocks

Sign-out should default to a button/form flow, not a GET link, matching modern
Devise/Turbo expectations.

## Deterministic Extraction

The DeviseHx generator should build an inventory before generating strong APIs.
Inputs:

- Bundler specs from `Gemfile` / `Gemfile.lock`
- installed Devise version and safe gem path
- Devise initializer literal config
- `rails routes` output
- `config/routes.rb` static parse for diagnostics
- model Ruby/Haxe source and literal `devise :...` declarations
- `db/schema.rb` preferred over historical migrations
- custom Devise controllers/mailers/helpers
- RBS/YARD when present
- RailsHx manifest and generated headers
- app test framework shape

Example inventory:

```json
{
  "version": 1,
  "gem": {"name": "devise", "version": "5.0.0"},
  "scopes": [
    {
      "scope": "user",
      "route_resource": "users",
      "model": "User",
      "table": "users",
      "modules": ["database_authenticatable", "registerable", "recoverable", "rememberable", "validatable"],
      "helpers": ["current_user", "user_signed_in?", "authenticate_user!"],
      "schema_status": "ok"
    }
  ],
  "unknowns": [],
  "unsafe_required": []
}
```

The draft JSON schema for this inventory is
[`railshx-devisehx-inventory.schema.json`](railshx-devisehx-inventory.schema.json).
It is intentionally conservative: uncertain dynamic config, inferred custom
controllers, missing schema data, and unsafe boundaries must be represented as
reviewable diagnostics instead of silently becoming typed APIs.

Fail closed for ambiguous scopes, dynamic `devise_for`, missing schema columns,
unknown custom Warden strategies, non-literal OmniAuth providers, missing
models, and unowned output files.

## Generator UX

Keep the generic lane:

```bash
bin/rails generate hxruby:adopt --gem devise --discover
bin/rails generate hxruby:adopt --gem devise --write contracts
```

Add a Devise-specific lane:

```bash
bin/rails generate devisehx:install User
bin/rails generate devisehx:adopt
bin/rails generate devisehx:controllers User
bin/rails generate devisehx:views User --hhx
bundle exec rake hxruby:devise:check
```

The generic lane remains good for quick app-local wrappers. The Devise-specific
lane owns stronger Devise semantics and reusable package behavior.

## LLM-Assisted Adoption

LLM assistance is allowed only after deterministic extraction:

1. Run deterministic discovery.
2. Emit inventory, generated contracts with TODO/review markers, and a prompt
   packet.
3. Let an LLM propose a patch.
4. Human reviews the patch.
5. Gates run: Haxe compile, Ruby syntax, route parity, schema/module
   validation, Rails runtime tests, HHX/template snapshots, and gitleaks.

LLM output must not delete review markers without deterministic evidence, add
unmarked `Dynamic`, invent scopes/modules/columns/helpers, or introduce raw Ruby
or raw ERB in canonical examples.

## Escape Hatches

| Boundary | Safe default | Escape hatch |
| --- | --- | --- |
| Unknown scope | generated `DeviseScope<T>` | `Scope.unsafe(name)` |
| Unknown module | typed `DeviseModuleSpec` | `DeviseModule.unsafeCustom(name)` |
| Warden | `Auth.signIn/signOut/current` | `WardenAccess.unsafeWarden` |
| Dynamic routes | Rails-owned route + route externs | `DeviseRoutes.uncheckedRuby` |
| Dynamic params | typed field refs | `DeviseParams.unsafePermit` |
| Existing ERB | HHX or checked adoption | explicit `--force` / unchecked template boundary |
| OmniAuth provider | literal provider enum | `OmniAuthProvider.unsafe` |
| Mailer tokens | opaque token abstracts | no canonical string escape |

Canonical examples should avoid unsafe APIs.

## Testing Plan

Required coverage:

- Haxe compile tests for one scope and multiple scopes.
- Negative compile tests for wrong scope, unknown module, missing module schema,
  unguarded `currentRequired` under strict mode, invalid lifecycle hooks, and
  invalid route/helper assumptions.
- Compiler snapshots for model `devise :...`, lifecycle auth filters,
  parameter sanitizer, custom controllers, HHX helpers, route externs, and
  inventory.
- Generator smoke tests for missing Devise, missing model, ambiguous scopes,
  missing schema columns, unsafe overwrite attempts, and generated contract
  compile.
- Rails request tests for protected route redirect, login, logout, signed-in
  access, and bad login.
- Playwright for guest/login/protected/sign-out browser flow once the dogfood
  app exists.
